===============================	VCS =========================
Version Control System -	«система контроля версий» —программное обеспечение для облегчения работы с изменяющейся информацией. Оно позволяет хранить несколько версий одного и того же документа, при необходимости возвращаться к более ранним версиям, определять, кто и когда сделал то или иное изменение.
Представим, что вы работаете над каким-то проектом и в вашей директории оказалось множество различных версий одного и того же файла с кодом который находится:  C:\Users\ibondar\Documents\Academy T1\Java. базовый курс
Сложно разобраться с тем, что здесь происходит, не правда ли?
Если вы работаете с кодом и текстовыми файлами, то наверняка сталкивались с проблемой хранения и использования разных версий.
Вопросы, которые всегда возникают по пути работы:
	-Как сохранить стабильную версию файла и проекта?
	-Что, если вы сохраните изменения, но вам потребуется вернуться к предыдущей версии?
	-Как разобраться в этих версиях, если их очень много?
	-Как посмотреть изменения между версиями?
	-Как удобно организовать совместную работу над проектом
Даже если вы работаете в одиночку, то можете столкнуться с этими проблемами. Если же вы — часть команды, то эти вопросы необходимо как-то решать.
patch	-разность между файлами. список различий между двумя версиями файла или набора файлов. 
-------------------------------	Централизованные VCS:
Используют центральный сервер для хранения всех версий файлов. Разработчики взаимодействуют с этим сервером, получая актуальные версии файлов и отправляя свои изменения. Принцип работы:
-Разработчики получают последнюю версию кода с сервера на свои локальные машины для изменений.
-После внесения изменений они фиксируют их в центральном репозитории, где разрешаются конфликты и объединяются обновления.
-В этой модели только один разработчик может изменять конкретный фрагмент кода за раз — система блокирует файл, чтобы другие разработчики не могли получить к нему доступ во время редактирования.
+центральный сервер(администратор сервера, история);
-централизованный-это точка отказа.
-------------------------------	Распределённые (удалённые) VCS:
Каждый разработчик имеет полную копию репозитория со всей историей изменений. Изменения можно вносить локально, а позже синхронизировать с версиями других разработчиков.
+более стабильный, чем централизованный;
+у каждого разработчика, локально, находится весь слепок, значит можно восстановить данные.

/////////////////////////////// Что же делать?
Для решения этих вопросов придумали системы контроля версий. Это программные решения, которые позволяют множеству разработчиков одновременно работать над проектом, сохранять версии файлов, просматривать и откатывать изменения, делать ревью изменений и многое другое.
Мы будем работать с самой популярной системой контроля версий для разработки ПО — 
-------------------------------	Git-Global Information Tracker:
Git 		распределённая система хранения версий для фиксации изменений в файлах. Она разработана для координации работы между программистами. В отличие от некогда популярных систем вроде CVS и Subversion (SVN), где полная история версий проекта доступна лишь в одном месте, в Git каждая рабочая копия кода сама по себе является репозиторием. Это позволяет всем разработчикам хранить историю изменений в полном объёме. И если с файлами основного репозитория что-то случится — проект легко восстановить из копии любого участника команды.

/////////////////////////////// Распределённая система — это хорошо?
Благодаря тому что Git распределённый, можно вносить изменения в код без доступа к интернету, локально, сохраняя все данные в папках на жёстком диске, которые называются репозиторием.
Чтобы иметь доступ к репозиторию с различных устройств, можно хранить копию репозитория онлайн, что сильно облегчает работу над одним проектом для нескольких людей. Для этой цели существуют специальные платформы хранилища кода, такие как GitHub.
-------------------------------	GitHub:
веб-сервис, который основан на системе Git. Его часто называют «социальной сетью для разработчиков», которая помогает удобно вести коллективную разработку IT-проектов. Здесь можно публиковать и редактировать свой код, комментировать чужие наработки, следить за новостями других пользователей. 
Создавая удалённый репозиторий в GitHub и выкладывая на него код, вы можете открыть возможность для совместного использования кода проекта другими разработчиками и для презентации ваших проектов.

/////////////////////////////// Рассмотрим пример
Разработчик Пётр получил задание и добавляет функцию для будущей версии 2.0, после чего отправляет изменения и сопровождает их описанием.
Затем он разрабатывает другую функцию и делает ещё несколько изменений в файлах проекта. И сохраняет отправляет их в Git. Эти изменения сохраняются в истории Git в виде отдельных рабочих элементов, к которым при потребности можно вернуться. Петру приходит срочная задача поправить баг в версии 1.3. Он переключается на ветку, соответствующую версии 1.3 того же ПО — там он сможет исправить баг, затрагивающий эту конкретную версию. Это нужно, чтобы команда Петра могла выпустить версию 1.3.1 с исправлениями до завершения работы над версией 2.0. После исправления бага Пётр вернётся к ветке для версии 2.0 и продолжит работу над будущими функциями. Все перечисленные действия можно выполнить даже без доступа к сети, поэтому система Git отличается быстротой и надежностью, даже если работать в самолете. Когда Пётр будет готов отправить все внесённые изменения ему останется лишь выполнить команду git push и внести изменения в локальный репозиторий, а потом отправить pull request и внести изменения в удалённый репозиторий.
