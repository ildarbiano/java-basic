========================== Объектно-ориентированное программирование:
ООП 			—парадигма программирования, в которой основными строительными блоками являются объекты. Объекты объединяют данные (состояние) и методы (поведение), которые работают с этими данными, в единое целое. Простыми словами: ООП позволяет моделировать реальный мир в коде, представляя сущности как «объекты» с характеристиками (данными) и поведением (функциями). 
ООП 			—парадигма, в которой программа строится из взаимодействующих объектов.
+	упрощение разработки. Разделение кода. Инкапсуляция.;
+	возможность создания масштабируемых систем. Работа с наследниками;
+	лёгкость модернизации, с сохранением обратной совместимости.
-	неэффективность на этапе выполнения;
-	неэффективность, в смысле распределения памяти;
-	излишняя избыточность;
-	психологическая сложность проектирования. Большое количество объектов;
-	техническая сложность проектирования и документирования. Большое количество объектов.
Объект			-мыслимая или реальная сущность, обладающая характерным поведением и отличительными характеристиками. Эта сущность является важной в предметной области.
Объект			-всё в java, кроме примитивных типов.
-------------------------- Структура ООП:
В коде, написанном по парадигме ООП, выделяют четыре основных элемента: 
Объект 			—часть кода, которая описывает элемент с конкретными характеристиками и функциями. 
Объект 			-экземпляр класса, который имеет одинаковый набор свойств;
Объект			-экземпляр класса, который имеет общее с классом поведение. 
Объект			-экземпляр класса, который имеет некоторое: состояние, поведение, уникальность. Например: машина имеет набор состояний: цвет, уровень топлива. машина ездит, сигналит, двери открываются. Каждая машина уникальна, например по серийному номеру.  
Class			—шаблон, на базе которого можно построить объект. Исполняемый код находится, только, здесь. 
Class			—описание созданного объекта. 
Class			—в классе задаются общие характеристики и действия, которые будут у всех объектов этого типа. Например, для интернет-магазина можно создать класс «Карточка товара», где будет заранее определено, что у каждой карточки будут следующие характеристики: название, цена и производитель.
Class			—контракт методов:
	class MyClass {
	}
Контракт класса		-набор методов и полей класса, открытых для доступа извне тем или иным способом, в совокупности с описание их назначения;
Контракт класса		-способ выражения обещания автора-разработчиа, относительно того, на что способен и для чего предназначен созданный им продукт.;
Class			—содержит модификаторы, поля и методы:
Class's поле		—состояние объекта. Если поле - примитив, то инициализируются со значением by default, также могут снабжаться модификаторами.
Class's method		-поведение класса. Можно переопределять, перегружать, также могут снабжаться модификаторами.
Class's конструктор	—метод класса, предназначенный для первоначального формирования объекта, при его создании.
	конструктор дочерних классов	-вызываются при создании объектов дочерних классов;
	конструктор дочерних классов	-могут вызвать друг друга по ключевому слову this();
	конструктор дочерних классов	-могут вызвать конструкторы "базового"; класса, по ключевому слову super(..)
	ключевое слово super(..)	-может не использоваться, только если в родительском классе существует конструктор by default.
Class			-стандартная библиотека java предоставляет большой набор классов. Можно создавать свои.
Class			-используется как параметр метода method;
Class			-используется как возвращаемое значение метода method;
Class			-используется как локальная переменная;
Class			-используется как поле класса, того же или другого класса;
Method			—функция внутри объекта или класса, которая позволяет взаимодействовать с ним или другой частью кода.
Атрибут			—характеристики объекта, например, цена, производитель или объём оперативной памяти. В классе прописывают, что такие атрибуты есть, а в объектах с помощью методов заполняют эти атрибуты данными.
МОДИФИКАТОР доступа	private		доступ только в пределах класса;
МОДИФИКАТОР доступа	protected	доступ в пределах класса, потомков класса, пакета - package;
МОДИФИКАТОР доступа	package-private	доступ в приделах package. Default;
МОДИФИКАТОР доступа	public		доступ для всех.

-------------------------- Принципы ООП:
В основе ООП лежат четыре ключевых принципа: 
Инкапсуляция 		—сокрытие внутреннего устройства и предоставление только необходимого внешнего интерфейса. Отделение внутреннего представления от внешнего.
Наследование 		—один класс (объект-шаблон) может наследовать свойства и поведение другого. Создание производных классов, наследующих свойства базового класса, что позволяет создавать более специализированные классы на основе общих.
Полиморфизм 		—разные объекты могут реализовывать одинаковые методы по-разному. Один интерфейс - множество реализаций, т.е. способность системы иметь одинаковое поведение, при различной реализации.  Это упрощает работу с обобщёнными интерфейсами и расширение функциональности.
Абстракция 		—сложные системы представляются в виде простых моделей, скрывающих ненужные детали. Программист оперирует понятиями «что делает», а не «как именно».

========================== НАСЛЕДОВАНИЕ и ПОЛИМОРФИЗМ:
Полиморфизм 			позволяет объектам разных классов отвечать на один и тот же вызов метода по-разному.
Полиморфизм 			позволяет объектам быть совместимыми в базовом и дочернем классе.
ПЕРЕОПРЕДЕЛЕНИЕ			позволяет объектам подклассов иметь собственную реализацию методов, определённых в суперклассе.
ПЕРЕОПРЕДЕЛЕНИЕ методов		фундаментальный механизм ООП, обеспечивающий полиморфизм.	
-------------------------- совпадение имён методов:
в родительском и дочернем классах.
сигнатуры различны	ПЕРЕГРУЗКА		-добавляется метод, с другими параметрами, к контракту доступа;
сигнатуры совпадают	ПЕРЕОПРЕДЕЛЕНИЕ		-замещение версии метода, объявленной в базовом классе, новой, с точно такой же сигнатурой.
			ПЕРЕОПРЕДЕЛЕНИЕ		-при обращении извне объекта, производного класса, к его методу - всегда, вызывается новая версия метода;
			ПЕРЕОПРЕДЕЛЕНИЕ		-доступ, к методу базового класса, только изнутри объекта, дочернего класса, может быть получен с помощбю ключевого слова super;

class Parent {
int number ;
Parent ( int number ) {
	this.number = number ;			// установка значения поля number
	}

	return this.number;
	}
}						//
		
class Child extends Parent {			// класс наследник extends
	Child (int number) {
		super(number);
	}
	@Override				//Аннотация в Java - указывает компилятору, что текущий метод переопределяет метод из суперкласса.

	int getNimber () {			
int number =
super.getNumber();				// переопределяющий метод, который будет ...
return Math.max(10,number);			// ... возвращать максимум из двух чисел
	}
}	
Parent e = new Parent (3);			// e. getNuber () -> 3	экземпляр по ссылке, является экземпляром класса Parent
e = new Child(10);				// e. getNuber () -> 10	экземпляр по ссылке, является экземпляром класса Child
Child d = ( Child ) e;				// d. getNuber () -> 10
@Override		-Аннотация в Java. С английского языка аннотация переводится как «переопределён». Наличие аннотации необязательно: если в дочернем классе создать метод с такой же сигнатурой, как у родительского, метод всё равно переопределится. Но рекомендуется ставить аннотацию всегда, так как это улучшает «читабельность» кода, а также при этом компилятор проверит на этапе сборки, что такой метод действительно есть в родительском классе.

-------------------------- SOLID:
аббревиатура, которая обозначает пять ключевых принципов проектирования классов в объектно-ориентированном программировании;
набор полезных принципов:
принципы помогают создавать понятный, гибкий и легко поддерживаемый код.;
принципы позволяют писать поддерживаемый и чистый, легко масштабируемый код;
принципы SOLID сформулировал американский инженер-программист Роберт С. Мартин. Позже, в 2004 году, консультант по разработке Майкл Физерс предложил объединить эти идеи под аббревиатурой SOLID;
принцип не панаця, а хорошая практика. Расшифровка аббревиатуры:
S 	Single responsibility 		(principle) принцип 	-единственной ответственности;
O 	Open-closed 			(principle) принцип 	-открытости/закрытости;
L 	Liskov substitution		(principle) принцип 	-подстановки Барбары Лисков;
I 	Interface segregation		(principle) принцип 	-разделения интерфейсов;
D 	Dependency inversion		(principle) принцип 	-инверсии зависимостей.
Применение SOLID позволяет создавать гибкую архитектуру, в которой каждый компонент приложения выполняет свою конкретную задачу, не вмешиваясь в работу других. Такой подход делает код легче в тестировании, поддержке и доработке, а изменения в одной части системы не приводят к непредвиденным проблемам в других модулях.

-------------------------- S-принцип-единственной ответственности:
-Метод должен иметь единственную обязанность, а не выполнять несколько функций: загрузка ДБ, сортировка, поиск максимума. 
-Каждый объект должен иметь одну ответственности, которая полностью инкапсулирована в классе. Всё поведение объекта должны быть направлены на обеспечение ответственности.
-Методы должны писаться принципу исходя из именования класса
-------------------------- O-принцип-открытости/закрытости:
-Объекты, классы, модули, функции, методы - должны быть открыты для расширения, но закрыты для изменения.
-------------------------- L-принцип-подстановки Барбары Лисков:
-Объекты, которые используют базовый тип должны иметь возможность использовать подтипы базового, не зная об этом.
-------------------------- I-принцип-разделения интерфейсов:
-"Толстые" интерфейсы нужно разделять на маленькие и специфические по: своим обязанностям, своему названию; чтобы объекты "маленьких" интерфейсов знали только методы, которые им необходимы для работы;
-Объекты не должны зависить от методов, которые они не используют;
-При использовании интерфейса не должны меняться классы, которые они не используют.
-------------------------- D-принцип-инверсии зависимостей:
-Модули верхних уровней не должны импортировать сущность из модулей нижних уровней;
-Абстракции не должны зависить от деталй, а детали должны зависить от абстракции;

========================== МЕХАНИЗМ ИСКЛЮЧЕНИЯ В java:
Exception в Java 		—это событие, которое нарушает нормальный ход выполнения программы. Проще говоря, это «сигнал» от JVM: что-то пошло не так во время выполнения кода. 
Исключения в Java 		-нужны для того, чтобы программа могла корректно реагировать на ошибки, не падала полностью и давала возможность продолжить работу или корректно завершиться. Некоторые примеры исключений:
FileNotFoundException		попытка открыть несуществующий файл;
NullPointerException		обращение к неинициализированной переменной;
NoSuchElementException		клик по несуществующему элементу.

КЛАССИФИКАЦИЯ EXEPTION:
---------Объявляемые, проверяемые - checked: 
Компилятор требует обязательно обработать или указать в сигнатуре метода. К ним относятся, например, IOException и SQLException. Чаще всего связаны с внешними ресурсами, такими как файлы, базы данных или сетевые соединения.
-носят предсказуемый характер;
-указываются в объявлении метода в коде или специальной для исключения синтаксической конструкции;
-наследуются от класса Exception.
---------Непроверяемые (unchecked):
Их можно не ловить, и компилятор не будет ругаться, но они часто приводят к падению программы, если их не предусмотреть. К таким исключениям относятся NullPointerException, IndexOutOfBoundsException и NoSuchElementException. Обычно возникают из-за ошибок в логике программы или тестов.
-обусловлены логикой кода;
-не указываются в объявлении метода, но могут быть указаны в конструкции для обработки исключения;
-наследуются от RuntimeException.

-------------------------- Stack в контексте exeptions:
—это структура данных, которая внутри виртуальной машины Java (JVM) используется для обработки последовательности действий при выполнении программы. Каждый вызов метода при выполнении программы записывается как элемент класса StackTraceElement. Когда выполнение метода завершается, информация о нём удаляется из стека. Если исключение прерывает работу программы, то один или несколько методов не могут завершиться, и информация о них остаётся в памяти JVM в виде массива StackTraceElement. Перед тем как завершить программу, JVM печатает содержимое этого массива. Такое сообщение об ошибке называется стек-трейс (stack trace) — «след стека». С его помощью можно определить, в какой части программы произошла ошибка, и узнать, как программа пришла к этому месту.

	-выполнить некоторые восстановительные действия;
	-выбросить собственное исключение;
	-осуществить необходимые действия и передать управление последующим инструкциям;
-количество блоков catch не регламентировано.
-предложения catch рассматриваются последовательно до обнаружения среди них того типа, который допускает присвоение выброшенного исключения;
-список предложений catch просматривается только один раз;
-лучше указывать самый узкий тип, дочерний, class exeptions, так как он более информативен, как минимум, за счёт своего названия;
-должны следоват от самого узкого, до самого широкого типа class exeptions.
------ блок finally:
-выполнится в любом случае:
	-при успешно выполнении try;
	-при выбрасывании exeption;
	-при передаче управления по break или return!
-не обязателен блок finally;
-если есть блок finally, то блок catch - необязателен;
-если присутствует блок finally, то выполняется после завершения работы остальных фрагментов кода try.
-------------------------- отлов exeptions нескольких типов:
-нужные типы exeptions разделяются | вертикальной чертой;
-блоков catch, с таким синтаксисом, может быть много;
-в блоке catch можно остольковать только общие методы, указанных типов exeptions;
-переменная, куда помещается ссылка на объект исключений, неявно приобретает модификатор final.
try {
	//некоторый код
}
catch (ClassNotFoundExeption | IOExeption ex) {
	logger.log(ex);
	throw ex;
}
-------------------------- выбрасывание exeptions:
-объявляемые и необъявляемые исключения, выбрасываемые вызываемыми методами и операторами;
-явно (принудительно) выбрасываемые exeptions:
throw referenceToThrowableObject;			//выбрасываем существующее исключение
throw new NoSuchAttributeExeption(name);		//создаём новое исключение
-------------------------- создание пользовательских exeptions:
-создаётся новый, узкого тип, наследующий от более шировкого, подходящего по смыслу, типа. например: java.lang.IndexOutOfBoundsExeption
-само то, что выбрасывается exeption более узкого типа, - несёт в себе информацию;
-в составе нового типа, могут входить новые поля и методы;
-чаще всего class exeptions содержит только два конструктора (по-умолчанию и с сообщением (параметром-строкой)), просто вызывающие конструкторы родительского class exeptions;
-современные среды разработки облегчают создание собственных class exeptions.
-------------------------- преимущества о использования exeptions:
+единая логика обработки ошибок. Обработка на любом уровне;
+разделение логики по обработки ошибок и бизнес приложения;
+необходимость обработки checked exeptions влечёт за собой действия по восстановлению приложения. например: если ошибка о невозможности записи информации в файл, которого нет, то лучше предусмотреть создание файла, тем самым обеспечить дружественность программы пользователю.
