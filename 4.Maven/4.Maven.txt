================== MANIFEST pom.xml JAR ==============================================
MANIFEST.MF лежит, В jar: META-INF/MANIFEST.MF
- ключевой компонент JAR-файла, который содержит метаданные, необходимые для правильного запуска приложения. Этот файл определяет точку входа в приложение, версию, зависимости и другую важную информацию. говорит JVM, что запускать. Некоторые области применения MANIFEST.MF:
	-Создание исполняемых JAR-приложений — определение основного класса для запуска приложения;
	-Управление зависимостями — указание внешних библиотек;
	-Конфигурация приложения — предоставление информации во время выполнения;
	-Безопасность и подписание — включение метаданных, связанных с безопасностью.
MANIFEST описывается ТОЛЬКО через pom.xml
MAVEN читает MANIFEST описанный через pom.xml всё, что должно попасть в JAR → pom.xml
maven-jar-plugin 		плагин в pom.xml создают описание манифеста для запуска архива приложения.
maven-shade-plugin		плагин в pom.xml создают описание манифеста для запуска архива приложения.

==================== Apache Ant ====================================
build.xml в Apache Ant — это XML-файл, который содержит инструкции для сборки и управления проектом. Он служит центральным файлом конфигурации для Ant и содержит указания на выполнение задач в процессе сборки:
<project>  
    <target name="clean">								//очистка от предыдущей сборки. ant clean
        <delete dir="${build.dir}"/>  							//удаление предыдущего, далее по коду это "build/classes"
    </target>  

    <target name="compile" depends="clean">  
        <mkdir dir="${build.dir}"/>  							//куда сложить скомпелированные файлы, далее по коду это "build/classes". ant complite
        <javac srcdir="src" destdir="build/classes"/>  					//где находятся java файлы проекта
    </target>  

    <target name="jar">  								//ant jar
	<mkdir dir="build/jar">
        <jar destfile="build/jar/myproject.jar" basedir="${build.dir}"/>  
		<manifest>
			<attribute name="Main-Class" value=""/>
		<manifest>
	</jar>
    </target>  

    <target name="run"> 								//ant jar
	<java jar="build/jar/myproject.jar" fork="true"/>
    </target> 
</project>

-------------------- pom.xml:
			- это XML-файл, который содержит объектную модель проекта (Project Object Model) в фреймворке Apache Maven. 
			- описывает структуру проекта, зависимости, плагины и конфигурации сборки, необходимые для компиляции, тестирования, упаковки и развёртывания приложения. 
			- параметры настройки сборки проекта.
Project Object Model	— это декларативное описание проекта, в котором указываются, например, координаты проекта (groupId, artifactId, version), зависимости, свойства и профили сборки. 
Effective POM 		— это концепция в Maven, которая означает объединение настроек по умолчанию из файла Super POM и конфигурации, определённой в файле POM проекта. 
Структура, некоторые разделы файла pom.xml:
	Информация о проекте 	— корневой элемент <project>. Содержит метаданные и конфигурацию, используемые Maven для управления процессом сборки.
	Координаты проекта 	— определяют уникальные идентификаторы проекта в репозитории Maven. Например: groupId — идентификатор группы проекта, artifactId — идентификатор проекта или модуля, version 	— текущая версия проекта.
	Раздел свойств 		— определяет многоразовые переменные для поддержания согласованности и упрощения обновлений версий.
	Раздел зависимостей 	— перечисляет все внешние библиотеки, на которые зависит проект. Каждая <dependency> включает groupId (организацию библиотеки), artifactId (имя библиотеки) и version (версию библиотеки).
	Раздел репозиториев 	— определяет удалённые репозитории, из которых Maven получает зависимости.
Крупные проекты могут быть поделены на несколько модулей (подпроектов), каждый со своим собственным POM. Операции над модулями могут выполняться через общий корневой POM единой командой. Содержание: в файле pom.xml также могут быть указаны, например:
	Описание проекта 	— имя проекта, URL-адрес, организация, список разработчиков и вкладчиков, лицензия проекта.
	Настройки сборки 	— <build> можно изменить местоположение исходного кода и тестов, добавить новые плагины, прикрепить цели плагинов к фазам жизненного цикла.
	Профили 		— альтернативные пути сборки проекта, разбитые по профилям.
строгая структура:	
<project>
    <modelVersion/>
    <parent>    </parent>	-как в ООП, вынести в родительский POM, например в Spring-boot проектах 

    <groupId/>
    <artifactId/>
    <version/>

    <properties/>		-свойства, для наследования и переменных

    <dependencies/>		-зависимости

    <build>			-сборка
        <plugins>		-плагин

-------------------- Maven Lifecycle Default (Build):
Обрабатывает стандартный процесс сборки проекта — от компиляции кода до упаковки и установки. Некоторые фазы этого жизненного цикла: 
Validate 	— проверяет конфигурацию проекта, чтобы убедиться, что вся необходимая информация присутствует и корректна.
Compile 	— компилирует исходный код проекта, но не создаёт архива.
Test 		— запускает единичные тесты с помощью фреймворка тестирования, чтобы убедиться, что код работает как ожидалось.
Package 	— упаковывает скомпилированный код в формат JAR, WAR или другой формат артефакта.
Verify 		— выполняет дополнительные проверки упакованного кода, чтобы убедиться, что он корректен.
Install 	— устанавливает упакованный артефакт в локальный репозиторий для использования как зависимость в других проектах.
Deploy 		— распределяет упакованный артефакт в удалённый репозиторий "remote" для обмена с другими разработчиками или командами.
Важно: жизненный цикл последовательен: 
mvn install	- Maven автоматически запустит validate, compile, test, package и verify сначала. 
-------------------- Clean:
Обрабатывает процесс очистки проекта — удаляет артефакты, сгенерированные в предыдущих сборках, чтобы обеспечить свежую среду сборки. Некоторые фазы этого жизненного цикла: 
pre-clean 	— выполняет процессы перед очисткой.
clean 		— удаляет файлы, сгенерированные в предыдущей сборке.
post-clean 	— выполняет процессы после очистки.
Важно: фазы жизненного цикла не могут выполнять задачи самостоятельно — Maven делегирует их плагинам. Например, фаза clean не удаляет директорию сборки, а передаёт эту задачу плагину maven-clean-plugin. 
-------------------- Site:
Генерирует документацию проекта — отчёты и другую информацию. Некоторые фазы этого жизненного цикла: 
pre-site 	— выполняет процессы перед генерацией документации сайта.
site 		— генерирует документацию сайта.
post-site 	— выполняет процессы после генерации документации сайта.
site-deploy 	— распределяет сгенерированную документацию сайта на веб-сервер.
Важно: при запуске одной фазы жизненного цикла site также выполняются все предыдущие фазы. Например, команда: 
mvn site 	- запускает фазы pre-site и site. 
Важно: стандартные жизненные циклы могут быть дополнены функционалом с помощью плагинов, но на практике это случается редко.
